# TDD Linear

[TDD-LINEAR MODE ACTIVATED]

> Smelter 워크플로우 Step 4 (TDD 설계) 에서 호출된다.
> 기존 `tdd` 스킬 + Superpowers TDD 규칙 + 커스텀 면제 조건을 통합한다.

---

## 철의 법칙

**테스트 없이 프로덕션 코드를 쓰는 순간 → 멈추고 코드를 삭제하라. 처음부터 다시.**

**기능 당 최소 10개 테스트** — 이보다 적으면 TDD가 완료되지 않은 것이다.

---

## 면제 조건 (TDD 불필요)

다음에 해당하면 테스트를 요구하지 않는다:

| 변경 유형 | 예시 | TDD 필요? |
|---------|------|---------|
| 디자인/스타일 변경 | CSS 클래스, spacing, color, typography, visual polish | ❌ |
| 번역/i18n 변경 | 텍스트 복사본, 리소스 파일 업데이트 | ❌ |
| 기존 기능 수정 | 로직 변경, 버그 수정 | ✅ (기존 테스트 먼저 확인) |

### 기존 기능 수정 시 절차

1. 관련 기존 테스트 검색
2. 수정으로 인해 실패해야 할 테스트 확인
3. 해당 테스트를 **먼저 실패하도록 수정** (RED)
4. 그 다음 구현 진행 (GREEN → REFACTOR)

---

## RED-GREEN-REFACTOR 사이클

### 1. RED — 실패하는 테스트 작성

```
- 다음 구현할 기능의 테스트를 먼저 작성
- 테스트 실행 → 반드시 실패해야 함
- 통과하면 테스트가 잘못된 것 → 수정
```

### 2. GREEN — 최소 구현

```
- 테스트를 통과시키는 최소한의 코드만 작성
- "하는 김에..." 금지
- 테스트 실행 → 반드시 통과해야 함
```

### 3. REFACTOR — 정리

```
- 코드 품질 개선
- 변경마다 테스트 실행
- GREEN 상태 유지
```

### 4. REPEAT

```
- 최소 10개 테스트가 될 때까지 반복
- 각 사이클은 하나의 동작/경계 조건을 다룸
```

---

## 최소 10개 테스트 기준

각 기능에 대해 다음 카테고리를 커버해야 한다:

| 카테고리 | 테스트 최소 수 | 예시 |
|--------|------------|------|
| 정상 동작 (happy path) | 2+ | 올바른 입력으로 정상 결과 |
| 경계 조건 (boundary) | 2+ | 빈 값, 최소/최대값 |
| 실패 케이스 (error path) | 2+ | 잘못된 입력, 에러 처리 |
| 엣지 케이스 (edge case) | 2+ | 예외 상황, 특수 문자, 동시성 |
| 통합 (integration) | 1+ | 다른 모듈과의 상호작용 |

---

## 강제 규칙

| 상황 | 조치 |
|------|------|
| 테스트 없이 코드 작성 | STOP → 코드 삭제 → 테스트 먼저 |
| 첫 실행에서 테스트 통과 | 테스트가 잘못됨 → 수정하여 FAIL 유도 |
| 한 사이클에 여러 기능 | STOP → 하나씩 분리 |
| Refactor 건너뜀 | 복귀 → 정리 후 진행 |
| 10개 미만 테스트 | 계속 — 10개 달성 전까지 완료 아님 |

## 출력 형식

```
## TDD Cycle: [기능명]

### RED Phase
테스트: [테스트 코드]
예상 실패: [어떤 에러가 날지]
실제 결과: [실행 결과 — 실패 확인]

### GREEN Phase
구현: [최소 코드]
결과: [실행 결과 — 통과 확인]

### REFACTOR Phase
변경: [정리한 내용]
결과: [테스트 여전히 통과]

---

## 진행 상황
완료: X / 10+ 테스트
```

## 기억

규율 그 자체가 가치다. 지름길은 TDD의 이점을 파괴한다.
